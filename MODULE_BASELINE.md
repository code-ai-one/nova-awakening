**审查日期**: 2026-04-17

**目标**: 建立质量基线用于指导后续改进决策

## A.1.1 词法分析器 · `编译器/词法/`

### 模块结构
| 文件 | 字节 | 职责 |
|------|------|------|
| `词法分析.nova` | 15707 | 主入口、标记类型定义、扫描器状态机 |
| `字符工具.nova` | 28237 | UTF-8字符分类/判断 |
| `字符串扫描.nova` | 11989 | 字符串字面量解析 |
| `数字扫描.nova` | 9571 | 整数/浮点/十六进制扫描 |
| `注释处理.nova` | 9866 | 单行/多行注释处理 |

### 算法评估
- **状态机**: 逐字节扫描 + 位置/行号/列号三元追踪
- **标记类型**: ~50种token（整数/浮点/字符串/标识符/运算符/关键字/符号）
- **中文支持**: 字节级扫描(`字节取`)，中文关键字编码在`标记常量.nova`中

### 优点
- 零分配设计（直接字节取代替子串+编码）
- 行号/列号精确追踪
- 多模块分工清晰

### 质量痛点
1. **标记类型编号硬编码**：`标_结束=0`, `标_整数=10`等硬编码常量分散
2. **错误恢复机制弱**：遇到无法识别的字节直接返回`标_错误=1`，无上下文报告
3. **Unicode边界处理**：`字节取`是字节粒度，跨越多字节UTF-8字符可能产生非法序列

### 改进优先级
- 🟢 低：已能自举，错误恢复可后续增强
- **Todo映射**: A.6错误诊断 + B.1中文关键字扩展时一并改进

## A.1.2 语法分析器 · `编译器/语法/`

### 模块结构
| 文件 | 字节 | 职责 |
|------|------|------|
| `解析器.nova` | 13168 | 解析器状态机、期望/匹配、错误恢复 |
| `语句.nova` | 7801 | 语句级解析（定义/可变/如果/当/函数） |
| `表达式.nova` | 5137 | 表达式解析（运算符优先级） |
| `节点定义.nova` | 7857 | AST节点类型常量 |

### 算法评估
- **技术**: 递归下降（RD）+ 运算符优先级解析
- **状态**: `[标记列表, 当前位置, 错误列表, 循环深度, 函数深度]`
- **EOF哨兵**: `EOF标记 = [0, 0, []]`避免越界
- **错误恢复**: `追加错误`不中断解析

### 优点
- 状态结构简洁（5元组）
- 循环/函数深度追踪支持break/continue/return语义
- 前瞻`解析器_窥视(p, n)`支持LL(k)

### 质量痛点
1. **无左递归处理**：递归下降对左递归表达式如`a-b-c`依赖手动展开
2. **错误消息粗糙**：`追加(p[2], [期望类型, 标记[0], 标记[2]])`只存类型码无上下文字符串
3. **无语法恢复策略**：遇到错误后继续解析但不跳到下一个同步点

### 改进优先级
- 🟢 低：基础功能完备，自举通过
- **Todo映射**: A.6错误诊断 + 未来可加入Panic-mode recovery

## A.1.4 IR生成器 · `编译器/规划/优化/SSA构建.nova`

| 尾调用.nova | 8906 | 尾调用优化 |

### 算法评估
- **IR设计**: SSA (静态单赋值) + φ函数
- **状态**: ⚠️ 实验性骨架，**未全量接入主编译流水线**
- **主链**: 使用 `IR/字节码/*`（Nova-Bytecode）路径

### 优点
- SSA设计现代（对标LLVM IR / GCC GIMPLE）
- 明确标记实验状态，不误导使用者

2. **对比器件混杂**：`SSA_整数常量=1`与`SSA_加=10`等数字范围不连续
3. **φ函数实现**：文件未完整读取，需要验证正确性

### 改进优先级
- 🟡 中：SSA激活能解锁大量优化，但非关键路径
- **Todo映射**: C.1原创IR代数+C.2优化算法的核心基础

## A.1.6 代码生成器 · `平台/代码生成/` ⚠️ 重点

### 模块结构
| 文件 | 字节 | 职责 |
|------|------|------|
| `x86_64/代码生成器.nova` | **118406** | AST→x86-64机器码主生成器 |
| `x86_64/x86编码器.nova` | 4077 | 寄存器操作/指令编码helper |
| `x86_64/PE生成器.nova` | 12069 | Windows PE格式 |
| `elf/ELF生成器.nova` | 470lines | Linux ELF格式 |
| `arm64/` | - | AArch64支持 |
| `riscv64/` | - | RISC-V支持 |
| `wasm/` | - | WebAssembly |

### 算法评估
- **主文件过大**: `代码生成器.nova` **118KB**（单文件最大）
- **职责**: AST→机器码编译，闭包/全局变量/尾调用/DWARF前置
- **多目标支持**: ELF/PE/MachO/Wasm/ARM64/RISC-V

### 核心发射函数举例
- `发射_载变量(disp)`: 处理栈局部+全局数组访问
- `发射_存变量(disp)`: 存储到局部/全局
- 依赖运行时`修正表`系统进行符号解析

### 质量痛点 ⚠️
1. **代码生成器单文件118KB**：单文件过大，难以维护，职责过多
2. **全局可变状态**：`代码生成器_目标ELF`, `当前编译模块名`, `闭包表`等全局变量污染
3. **⚠️ 内联strcmp根因地**: 编译字典访问时可能生成内联strcmp（A.2.3-A.2.5已源码根治）
4. **⚠️ heap残留根因地**: ELF零填充patch的workaround源于此模块的heap管理不确定性

### 改进优先级
- 🔴 **高**：代码生成器是**技术债根因地**，也是性能/正确性关键路径
- **Todo映射**: 
  - A.2.8根治零填充patch
  - A.5.8 代码生成器优化（指令缓存）
  - C.3 原创代码生成算法

# 总结

## 整体质量
- **正确性**: ✅ 自举稳定（fixed_point=true, diff_bytes=0）
- **完整性**: ✅ 8模块+智能优化+多平台codegen齐全
- **原创性**: 🌟 智能优化子系统/模块图编译器超越传统设计

## 关键痛点排名
1. 🔴 **代码生成器.nova (118KB)**: 单文件过大+全局状态+heap残留根因
2. 🔴 **模块图编译器.nova (163KB)**: 单文件过大+内联系统调用
3. 🟡 **符号表线性查找**: O(n)影响大项目性能
4. 🟡 **SSA未主链化**: 预留但未激活
5. 🟢 **错误诊断基础化**: 消息简陋无修复建议
6. 🟢 **链接器无增量**: 整体重链
