### 1-R1: SENTINEL 规范化智能跳过
- **根因**：所有条件表达式都发射 9字节 SENTINEL归零 + 3字节 test，但比较/逻辑运算结果必定 0/1
- **修复**：`表达式_已是布尔(节)` 辅助函数，`当`/`如果` 智能跳过
- **效果**：fib35 Nova/C 4.20x → 3.51x (-16%)，collatz 2.21x → 1.80x (-19%)
- **产物**：Stage2/3 7,909,376 → 7,643,136 (-266KB, -3.4%)

### 1-R2: cmp+jCC 条件融合跳转
- **根因**：比较+条件跳转走 `cmp + setCC + movzbl + test + je` 5指令链，可融合为 `cmp + jCC` 2指令
- **修复**：`编译_条件融合跳转(_cf节)` 辅助函数，整数比较直接生成 cmp+jCC，右操作数为立即数时用 `cmp rax, imm32` 避免 xchg
- **效果**：fib35 3.51x → 3.28x，collatz 1.80x → 1.62x
- **产物**：7,643,136 → 7,528,448 (-115KB, -1.5%)

## 4. 性能诊断（诚实评估）

### 4.1 Nova 目前落后 GCC-O2 1.62-3.28x 的原因
1. **无寄存器分配** — 局部变量全在栈上，每次访问都走内存
2. **无全函数寄存器级调用约定** — 参数压栈，返回值仅通过 rax
3. **每次 `i = i + 1` 都 load+store** — 无 live-range 分析
4. **字典操作走 runtime blob** — 非 inline, 每次 `字典取` 有函数调用开销

### 4.2 已修复的两类冗余（R1+R2）
1. **SENTINEL 规范化**（12字节/比较）：已智能跳过
2. **比较-跳转链**（3-8字节/比较）：已 cmp+jCC 融合

### 4.3 下阶段性能路线图
- **R3 候选**：局部变量寄存器保留（需 liveness analysis）
- **R4 候选**：函数参数通过寄存器传递（System V AMD64 ABI）
- **R5 候选**：字典操作路径 inline / 零参数函数内联

### 6.2 去 Rust 三步策略
1. **第一步 B**：用 Nova 重写 backend_seed.rs 的前端+codegen（估期 3-5 日）
2. **第二步**：让 Rust 种子退化为纯字节码搬运（~200行，只做 fs::read/fs::write + ELF 构造）
3. **第三步**：用 Nova 源码生成 Stage-0 的字节码产物，Rust 种子完全消失（Stage-0 变为 Nova 自举的字节码 bundle）
