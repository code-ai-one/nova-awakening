## 二、真实性能瓶颈 (基于本轮实测)

### 瓶颈 1: 递归性能 (fib35 3.8x 慢)
**原因**: 纯栈模型, 每次 call 都 push/pop, 无寄存器分配
**解决**: Linear Scan 寄存器分配 (r12-r15 保存热变量)
**预期**: 1.5-2x 提升 → Nova/C 降至 2x

### 瓶颈 2: 简单循环 (loop_sum 2.2x 慢)
**原因**: 每次变量访问走栈 [rbp - N*8]
**解决**: 循环变量直接分配寄存器 (LICM + RegAlloc)
**预期**: 接近 C 性能

### 瓶颈 3: 浮点运算 (推测 matvec 16x 慢)
**原因**: 纯标量 SSE, 未向量化
**解决**: L2 SIMD 循环向量化 (已有 SIMD intrinsic 基础)
**预期**: 2-4x 提升

## 三、全模块深度升级路线 (按优先级)

### P0: 立即提升 Nova/C 性能 (1-3 月工程)

#### 3.1 寄存器分配器 (Linear Scan)
**现状**: `编译器/规划/分配/` 骨架就位  
**方案**: 
- 生命周期分析 `def_pos → last_use_pos` (2 周)
- 热度评分 (循环内变量 10x 权重) (1 周)
- Linear Scan 分配 r12-r15 / xmm2-xmm15 (3 周)
- 集成到 `发射_载变量` / `发射_存变量` (3 周)
**预期**: fib35 从 3.8x → 1.5-2x 慢

#### 3.2 L2 SIMD 循环向量化
**现状**: L1 intrinsic 7 原语就绪, L2 循环识别骨架  
**方案**:
- 识别归约模式 `sum += a[j]` → horizontal reduction
- 识别元素模式 `c[j] = a[j] + b[j]` → packed ops
- 识别 AXPY 模式 `c[j] = c[j] + k*a[j]` → FMA
**预期**: matvec 从 16x 慢 → 2-4x 慢

#### 3.3 LICM 自动激活
**现状**: `编译器/分析/循环不变量提升.nova` 已实现, 未被主流水线调用  
**方案**: 在 `编译_处理_节_当` 入口调用 LICM 分析
**预期**: 循环内不变量自动提升, loop_sum 5-10% 提升

### P1: 内存效率极致化 (2-4 月)

#### 4.1 列表紧凑布局
**现状**: 每个列表元素 16 字节 (type + value)  
**方案**: 同质列表 (全 i64 / 全 f64) 紧凑布局 8 字节/元素
**预期**: 内存 50% 降, cache 命中率 2x

#### 4.2 字符串内部池
**现状**: 字符串字面量重复存储  
**方案**: 编译期去重, 运行时共享只读段
**预期**: binary size 10-20% 降

#### 4.3 GC 增量式
**现状**: 有 GC 基础设施 (`运行时/垃圾回收/`)  
**方案**: 增量式三色标记, STW 限制 < 1ms
**预期**: 长期服务可用

### P2: 架构层面重构 (月级)

#### 5.1 解耦 6 组循环依赖
**方案**: 抽取纯接口层 (常量/类型定义), 实现层依赖接口, 接口不依赖实现
**收益**: 模块可独立测试/替换

#### 5.2 SSA 优化管线激活
**现状**: `IR/优化器/SSA构建.nova` 完整实现, 未默认启用
**方案**: 主编译流水线启用 SSA pass (常量传播/DCE/CSE/GVN)
**预期**: 复杂代码 10-20% 提升

#### 5.3 DWARF 调试信息集成
**现状**: `编译器/分析/DWARF调试信息.nova` 框架就位
**方案**: 发射 DWARF 段, gdb 集成
